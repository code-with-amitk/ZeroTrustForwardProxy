== Modes

=== 1. SSL Decrypt (Inspection)
The proxy terminates the TLS, decrypts the traffic to inspect for malware, and then re-encrypts it using its own certificate to send to the client.

=== 2. SSL Do Not Decrypt (SSL-DND)
In SSL-DND mode, the proxy operates as a transparent Layer 4 tunnel. It utilizes the standard HTTP CONNECT method to establish a bi-directional byte-stream between the client and the destination without terminating the TLS session.

==== Key Characteristics:

* Zero Visibility: The proxy does not possess the destination's private keys and cannot inspect the encrypted payload.

* End-to-End Integrity: The TLS handshake occurs directly between the client and the remote server, ensuring that certificate validation and encryption remains unbroken.

* SNI-Based Routing: While the payload is opaque, the proxy uses the Host header from the CONNECT request (and optionally the SNI in the Client Hello) to perform DNS resolution and routing.

==== Implementation Flow (The "Bridge" Logic)

Adding a section on the internal "Next-Hop" logic clarifies how the proxy handles the two distinct TCP legs.

|===
|Phase|Description
|Inbound Leg|Proxy accepts a TCP connection on the listener port and parses the initial HTTP CONNECT header.
|Resolution|Proxy performs an asynchronous DNS lookup for the requested target.
|Outbound Leg|Proxy initiates a secondary TCP handshake with the target IP on the external interface (eth0).
|Promotion|Upon successful upstream connection, the proxy returns 200 Connection Established and promotes the sockets to a bi-directional "pipe."
|===

The proxy sees the CONNECT or the SNI (Server Name Indication) in the Client Hello, identifies it as "Trusted"  (e.g., Financial or Health sites), and stops acting as a Layer 7 gateway. It drops down to Layer 4 (TCP) and just moves segments back and forth without looking at the payload, ie like a blind pipe.
